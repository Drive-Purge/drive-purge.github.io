Security Architecture

A zero-compromise, local-first design. Read the exact technical implementation details of how we protect your Google Drive credentials and files.

Last Updated: May 22, 2026

Security-First Principles

Drive Purge was designed from day one as a zero-trust productivity app. Many storage cleaners upload file indexes, logs, and account tokens to their remote servers to process them. This exposes your Google Drive metadata to third-party databases, leaks credentials, and puts your business files at risk of security breaches.

Drive Purge adopts a strict **local-first architecture**. Every single code segment, database transaction, indexer scan, and cryptographic verification runs completely on your machine. Our code connects exclusively to your local loopback address (127.0.0.1) and directly to Google's official endpoints. We maintain zero remote application databases and gather no diagnostic telemetry.

1. Local Loopback & Loop Network Isolation

Drive Purge hosts a high-performance **FastAPI desktop backend** compiled directly into a native executable. To prevent remote network exploitation, the application's network layers are isolated:

  • Local Loopback Binding: The application server binds exclusively to the local loopback port 127.0.0.1:7823. It is physically impossible to access the application server from other computers, local Wi-Fi, or public subnets.
  • Zero Telemetry Outbound: Drive Purge has no analytics tracking, no error reporters (no Sentry, Mixpanel, or Amplitude), and no telemetry scripts. Outbound requests are limited only to Google API endpoints.

2. Cryptographic Storage & Database Security

All database transactions are handled locally via an SQLite engine running in high-performance Write-Ahead Logging (**WAL**) mode (PRAGMA journal_mode=WAL). Because standard SQLite databases store values in plaintext, we enforce **Fernet Application-Layer Encryption** on all sensitive credential fields.

🛡️ Application-Layer Credential Encryption Flow

Before writing to the local drivepurge.db file, the application takes sensitive fields like access_token and refresh_token, encrypts them using Fernet AES-128 symmetric key encryption in CBC mode, and base64-encodes the ciphertext. The SQLite file itself is stored in the user's home folder and is completely unreadable without the cryptographic key.

The Key-Lock Lifecycle

  • Key Location: The encryption key is stored at ~/.drivepurge/secret.key (Linux/macOS) or %APPDATA%\Drive Purge\secret.key (Windows).
  • Unix Permission Hardening: On Linux and macOS systems, the application sets strict POSIX permissions (chmod 600), meaning only the currently logged-in desktop user has read/write privileges.
  • Disaster Recovery: If the key file is lost or deleted, the encrypted columns are rendered unreadable. In this scenario, no file data is ever lost; the user simply re-authenticates to acquire a fresh set of tokens.

3. Google OAuth 2.0 & Token Revocation

Drive Purge never asks for, sees, or handles your Google account password. All authentication is managed via Google’s OAuth 2.0 flow:

  1. The application triggers your system's default browser to open Google's official authorization servers (https://accounts.google.com/o/oauth2/auth).
  2. Upon approval, Google redirects the session with an authorization code back to your local machine: http://127.0.0.1:7823/auth/callback.
  3. The desktop client performs the final secure token exchange at https://oauth2.googleapis.com/token and stores the credentials using Fernet encryption.

Instant Token Evacuation: When you click "Disconnect" in the settings panel, Drive Purge instantly sends a revocation request to Google’s servers (https://oauth2.googleapis.com/revoke) and permanently wipes all encrypted credential tokens from the local SQLite database.

4. Data Isolation & File Safety

Drive Purge interacts with your files via the official Google Drive v3 REST API. We implement multi-layered safety mechanisms to protect your cloud storage:

  • Metadata-Only Retrieval: During scans, Drive Purge only queries metadata fields (like id, name, mimeType, size, createdTime, modifiedTime). The actual contents of your documents are never scanned, downloaded, read, or cached.
  • Safe Trashing & Restore: Standard cleanup operations send files to your official Google Drive Trash folder (trashed: true) rather than permanently deleting them. This gives you a 30-day window to change your mind and allows for our One-Click Undo restore functionality (trashed: false).
  • Smart API Pacing: To avoid account restrictions, our scanning engine integrates a customizable rate-limit spacer (defaulting to 50ms delay per request) and includes dynamic exponential backoff retries to handle Google API 429 (Rate Limit Exceeded) statuses automatically.

5. Offline Cryptographic licensing

To enforce privacy, Drive Purge is one of the few SaaS products that **does not phone home to verify your Pro license**. We use public-key cryptography to perform local licensing validation.

🔑 Offline Ed25519 Signature Verification

When you load a license file, the application parses the JSON payload and verifies its authenticity locally using the Ed25519 public-key signature scheme (via Python's pynacl library). It re-serializes the payload and validates the cryptographic signature against our embedded hardcoded public key: 9xI8PCMOUeOsQeVsA38xrSSzXwL7SD0JVZPflODnkmU=

Stable Machine Fingerprint Lock

To prevent commercial license sharing, licenses are locked to your machine. Drive Purge generates a stable, privacy-compliant hardware fingerprint using a SHA-256 hash of your hardware components:

Fingerprint = SHA256(CPU Processor + MAC Address/Node + OS Platform + CPU Arch + OS Release + Hostname)[:16]

The application only checks this fingerprint locally; it is never shared, sent, or uploaded to our servers, ensuring your hardware identity remains completely private.

Security FAQ

Does anyone else have access to my Google Drive?

No. Every transaction goes strictly between your local desktop computer and Google's servers. No third parties (including us) have access to your OAuth tokens, database, or scanning logs.

Can Drive Purge delete files without my permission?

No. Drive Purge operates exclusively as a search and flag engine first. Deletions and trash movements are only executed when you explicitly select items and click the "Trash" or "Delete" buttons in the UI.

What happens if my database file gets stolen?

The SQLite database file `drivepurge.db` contains your OAuth access and refresh tokens. However, these are thoroughly encrypted with AES-128 via Fernet. An attacker would need access to your home directory key file (`secret.key`) to decipher them. If you suspect an intrusion, simply visit Google Account Permissions and revoke "Drive Purge" to immediately kill all active credentials.

How does the local license check work without internet?

Your license file contains an Ed25519 digital signature signed by our master private key. The application uses our hardcoded public key to mathematically verify that the license was issued by us and matches your machine fingerprint. Because this verification is purely mathematical, it requires zero active network connections.
