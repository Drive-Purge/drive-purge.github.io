About Drive Purge

The story behind the intelligent Google Drive cleanup tool trusted by professionals worldwide.

Our Mission

Google Drive has become the backbone of digital storage for millions of individuals and organizations. Yet as our digital lives expand, so does the invisible clutter — duplicate files accumulating across shared folders, oversized media files silently eating into storage quotas, orphaned documents left behind when parent folders are deleted, and empty directories creating a maze of abandoned hierarchy. Most users don't realize how much storage they are wasting until they receive the dreaded "Storage is full" notification.

Drive Purge was built to solve this problem once and for all. Our mission is to give every Google Drive user — from individual freelancers to enterprise IT teams — the tools to understand exactly what is consuming their storage, identify files that can be safely removed, and execute cleanup operations with confidence. We believe that storage management should be intelligent, transparent, and above all, safe.

Unlike cloud-based alternatives that require you to grant third-party servers access to your files, Drive Purge runs entirely on your local machine. We fundamentally believe that your files are your business. Our application communicates only with Google's official Drive API — no intermediary servers, no analytics endpoints, no telemetry. This local-first architecture isn't just a feature; it's a core design principle that shapes every engineering decision we make.

The Technical Foundation

Drive Purge is built on a modern, asynchronous Python stack designed for performance and reliability. At its core, the application uses FastAPI — one of the fastest Python web frameworks available — running on Uvicorn, an ASGI server built on uvloop for high-throughput async I/O. This backend serves both the API endpoints and the HTML interface to a PyWebview desktop window, creating a native-feeling application without the overhead of Electron.

The frontend communicates with the backend through a clean RESTful API architecture organized into purpose-specific route modules: authentication routes handle OAuth flow management, scan routes manage the scanning lifecycle with real-time Server-Sent Events (SSE) for progress updates, metadata routes serve application configuration and help content, license routes manage subscription validation, and UI routes render the Jinja2-templated HTML views. This modular design ensures that each concern is isolated and independently testable.

Data persistence is handled through aiosqlite, an async wrapper around SQLite that allows non-blocking database operations without sacrificing the simplicity and portability of SQLite's single-file database format. The database uses WAL (Write-Ahead Logging) mode and enforces foreign key constraints, providing both performance and data integrity. All scan results, flagged files, user preferences, and encrypted OAuth tokens are stored in a single drivepurge.db file in the user's home directory.

Six Intelligent Scan Modes

The heart of Drive Purge is its scanning engine, implemented in the DriveScanner class. When you initiate a scan, the engine executes a multi-phase analysis pipeline that first fetches all non-trashed files from your Google Drive, stores them in a temporary database table for efficient querying, and then runs up to six detection passes — each specialized for a different type of storage waste.

🔍 Duplicate Detection

Identifies duplicate files by matching on name, file size, or a combined name+size key. The engine groups files by the selected match criteria, keeps the most recently modified version, and flags all older copies for review. Each flagged file includes metadata about which file will be kept and why.

📦 Large File Finder

Surfaces files exceeding a configurable size threshold (default: 10 MB). Google Workspace files (Docs, Sheets, Slides) are automatically excluded since they don't consume storage quota. Results are sorted by size in descending order for maximum impact.

📅 Old File Scanner

Finds files that haven't been modified within a configurable time period (default: 90 days). Uses precise timestamp comparison against each file's last modification date. You can set thresholds of 90, 180, or 365+ days depending on your retention policy.

👻 Orphan Detection

Discovers files whose parent folders have been deleted or are no longer accessible. These "ghost files" are invisible in the Drive UI but still consume quota. The engine checks every file's parent chain against the fetched folder tree and the user's root folder ID.

📂 Empty Folder Cleanup

Lists all folders that contain no files or subfolders. Excludes the root folder and any folders explicitly marked in the exclusion list. Cross-references every folder ID against the parent references of all scanned files.

📋 All Owned Files

An optional comprehensive mode that flags every file you own for manual review. Useful for users who want to audit their entire Drive content, export a full inventory, or make bulk decisions about file retention.

How the Scan Engine Works

The scanning process begins with the file fetch phase, where the engine paginates through the Google Drive API's files.list endpoint. Each page can return up to 1,000 files (configurable via page_size in ScanConfig). The engine constructs a query that excludes trashed files and optionally excludes Google Workspace MIME types. Fetched files are parsed into normalized rows and bulk-inserted into a temporary scanned_files table using INSERT OR REPLACE operations with indexed columns for efficient subsequent queries.

After fetching completes, the engine enters the analysis phase, running each enabled detection pass sequentially. Each pass queries the scanned_files table using optimized SQL — for example, the large file detector uses a simple WHERE size_bytes >= ? query against an indexed column, while the duplicate detector uses GROUP BY ... HAVING COUNT(*) > 1 to identify file groups before fetching individual duplicates. A deduplication set (already_flagged_ids) ensures that files flagged by an earlier pass are not double-counted in later passes.

Throughout both phases, the engine emits progress events through a callback function, enabling the frontend to display real-time updates via Server-Sent Events. The scan can be cancelled at any time by calling the cancel() method, which sets an internal flag checked at the beginning of each phase and page fetch iteration. Upon completion or cancellation, the temporary scanned_files table is cleaned up, and final results are persisted to the flagged_files table with full metadata including file names, sizes, MIME types, parent folder information, flag reasons, and detailed diagnostic data.

The Cleaning Engine

Once files are flagged, the DriveCleaner class handles safe deletion. Files can be moved to Trash (reversible) or permanently deleted (irreversible), with all operations executed through the Google Drive API's files.update and files.delete endpoints. The cleaner supports both individual file operations and batch processing, where files are grouped into chunks of 10 and processed concurrently using asyncio.gather with configurable delays between chunks to respect rate limits.

Every operation includes automatic retry logic with exponential backoff. Rate-limit errors (HTTP 429 and certain 403 responses) trigger up to 3 retries with increasing wait times (1, 2, 4 seconds). Permission errors are caught and surfaced with descriptive messages explaining why a file could not be trashed or deleted. Files that return 404 (already deleted) are treated as successful operations. All actions are recorded in the database with timestamps, enabling the Undo feature to restore trashed files by calling files.update with {"trashed": false}.
